# Internationalization (i18n) — Design

**Issue:** #40 — feat: Internationalization
**Date:** 2026-06-10
**Status:** Approved

## Goal

Add internationalization support to the OpenChore frontend so the UI can be
translated into any language. This PR delivers the i18n *infrastructure*, a
language selector, browser auto-detection, English fallback, and a complete
English translation file extracted from the current hardcoded strings. The
actual French translation is left to the community contributor who requested
this feature.

## Decisions

| Decision | Choice | Rationale |
| --- | --- | --- |
| Preference scope | App-level, stored in `localStorage` | No backend/DB changes. Works on login/ambient screens that have no logged-in user. Auto-detection + persistence share one mechanism. |
| Library | `react-i18next` (+ `i18next`, `i18next-browser-languagedetector`) | Industry standard. Interpolation, pluralization, browser detection, fallback all built in. Translators edit plain JSON. |
| Translation content in this PR | English baseline + selector only | The French translation was offered by the contributor; we ship the structure they plug into. |

## Architecture

### 1. Initialization — `web/src/i18n/index.ts`

Initializes the i18next singleton:

- `resources`: imported translation JSON keyed by language code.
- `fallbackLng: 'en'` — required English fallback when a key or language is missing.
- `supportedLngs: ['en']` (plus any future codes) — unknown browser locales fall
  back to English cleanly.
- Detection via `i18next-browser-languagedetector` with order
  `['localStorage', 'navigator']` and `caches: ['localStorage']`, using the key
  `openchore_lang`. This is both the browser auto-detection and the persistence.
- `interpolation.escapeValue: false` (React already escapes).
- `returnNull: false`.

`web/src/main.tsx` imports `'./i18n'` once before `ReactDOM.createRoot(...)`.
No React provider is required — `react-i18next` reads the initialized singleton;
`useTranslation()` subscribes components to language changes.

### 2. Language registry — `web/src/i18n/languages.ts`

A single source of truth:

```ts
export interface LanguageOption { code: string; label: string; }
export const LANGUAGES: LanguageOption[] = [
  { code: 'en', label: 'English' },
  // contributor adds: { code: 'fr', label: 'Français' },
];
```

The selector, `supportedLngs`, and `resources` all derive from this list so adding
a language is a one-line change plus a JSON file.

### 3. Translation files — `web/src/i18n/locales/<code>/translation.json`

- Single `translation` namespace.
- Keys nested by area: `common.*`, `profile.*`, `dashboard.*`, `admin.*`,
  `setup.*`, `ambient.*`, `reports.*`, `photo.*`.
- Dynamic values use i18next interpolation: `"greeting": "Hi, {{name}}"`.
- Counts use i18next plural suffixes: `"points_one": "{{count}} point"`,
  `"points_other": "{{count}} points"`.

`en/translation.json` is populated by extracting every current hardcoded
user-facing string.

### 4. Language selector — `web/src/components/LanguageSelector/`

- `LanguageSelector.tsx` + `LanguageSelector.module.css`.
- Renders a `<select>` (or styled dropdown) populated from `LANGUAGES`.
- Current value = `i18n.resolvedLanguage`; `onChange` calls
  `i18n.changeLanguage(code)`, which persists to localStorage and re-renders.
- Placement:
  - **ProfileSelection** page (login/landing) — the no-user entry point everyone sees.
  - **AdminDashboard** settings area — so an admin can set the device default.

### 5. String extraction (the bulk of the work)

Replace hardcoded JSX text and string literals across `web/src/pages/**` and
`web/src/components/**` with `t('area.key')` via the `useTranslation()` hook.
`aria-label`, `placeholder`, `title`, and toast/error messages are included.
Strings that come from the backend/API (chore titles, user names, config-driven
content) are **not** translated — only static UI chrome.

## Testing

- **Vitest** — `web/src/i18n/i18n.test.ts`:
  - Unknown locale resolves to `en` (fallback).
  - A known key resolves to its English string.
  - Interpolation substitutes a variable.
  - Plural selects `_one` vs `_other`.
- **E2E (Playwright)** — must stay green unchanged. `navigator.language` in CI
  Chromium resolves to `en`, so English remains the default and existing
  text-based assertions still match. No e2e changes expected; if any assertion
  depends on a string we rekey, update the English JSON value to match exactly.
- **Type check** — `tsc` must pass; translation JSON imported as a typed module.

## Out of scope (YAGNI)

- Per-user language stored in the database.
- Locale-aware date/number/currency formatting.
- Right-to-left (RTL) layout support.
- The French (or any non-English) translation content.
